Theme Management

Theme structure, inheritance, and deployment

Theme Structure

Themes are components that personalize the storefront or Admin UI.

app/design/frontend/<Vendor>/<theme>/
├── <Vendor>_<Module>/
│ ├── web/
│ ├── layout/
│ └── templates/
├── web/
│ ├── css/
│ ├── js/
│ └── images/
├── media/
├── i18n/
├── theme.xml
├── registration.php
└── composer.json
  • theme.xml - declares theme name, parent theme
  • registration.php - registers theme with Magento
  • Default themes: Blank & Luma

🎯 File Types & Folder Structure - Exam Critical!

Know where each file type belongs

templates/ - PHTML files (PHP + HTML templates): 
<theme_dir>/<Vendor>_<Module>/templates/
└── custom.phtml

Example: app/design/frontend/Acme/theme/Magento_Catalog/templates/product/view.phtml

Content: PHP template files rendered by block classes
layout/ - Layout XML files: 
<theme_dir>/<Vendor>_<Module>/layout/
├── default.xml
├── catalog_product_view.xml
└── override/
    └── base/
        └── catalog_product_view.xml

Content: XML files defining page structure (blocks, containers)
web/css/ - LESS/CSS files: 
<theme_dir>/web/css/
├── source/
│   ├── _extend.less       // Extend parent theme
│   ├── _theme.less        // Custom theme styles
│   └── _variables.less    // Override LESS variables
├── styles-m.less          // Mobile entry point
└── styles-l.less          // Desktop entry point

Content: LESS files (compiled to CSS)
web/js/ - Theme JavaScript (vanilla JS, custom libraries): 
<theme_dir>/web/js/
├── theme.js
└── custom-library.js

Content: Plain JavaScript, jQuery plugins, 3rd party libraries
NOT for RequireJS modules or Knockout components (those go in module/view)
Module view/frontend/web/js/ - RequireJS & Knockout: 
<module_dir>/view/frontend/web/js/
├── view/
│   └── custom-component.js    // Knockout UI components
├── model/
│   └── data-model.js
└── custom-widget.js           // jQuery widgets

Content: RequireJS AMD modules, Knockout.js components, UI components
Used with x-magento-init
web/images/ - Theme images: 
<theme_dir>/web/images/
├── logo.svg
└── icons/
    └── cart.svg

Content: Theme-specific images (logos, icons, backgrounds)
media/ - Theme preview: 
<theme_dir>/media/
└── preview.jpg

Content: Theme preview image shown in admin
i18n/ - Translation files: 
<theme_dir>/i18n/
├── en_US.csv
└── de_DE.csv

Content: Translation CSV files for theme-specific strings
🔥 Quick Reference:
  • .phtml → templates/ folder
  • .xml (layout) → layout/ folder
  • .less/.css → web/css/ folder
  • Knockout.js → module/view/frontend/web/js/view/
  • RequireJS modules → module/view/frontend/web/js/
  • Theme JS → theme/web/js/
  • Images → web/images/
  • Translations → i18n/

Theme Inheritance

Themes can inherit from parent themes. Child themes override or extend parent files.

theme.xml example: 
<theme xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" 
       xsi:noNamespaceSchemaLocation="urn:magento:framework:Config/etc/theme.xsd">
    <title>Custom Theme</title>
    <parent>Magento/blank</parent>
</theme>

Fallback order: Custom Theme → Parent Theme → Module default

Theme Configuration

registration.php: 
use Magento\Framework\Component\ComponentRegistrar;

ComponentRegistrar::register(
    ComponentRegistrar::THEME,
    'frontend/Vendor/theme',
    __DIR__
);

Activate via Admin: Content → Design → Configuration

Static Content Deployment

bin/magento setup:static-content:deploy
bin/magento setup:static-content:deploy en_US -t Vendor/theme
bin/magento setup:static-content:deploy -f  # force
  • Generates static files in pub/static/
  • Production mode requires deployment
  • Developer mode generates on-the-fly

🎯 Fallback Logic - Critical for Exam!

Magento uses a fallback mechanism to locate theme files. Understanding this is essential.

⚡ Fallback Order for Templates & Layouts: 
1. <theme_dir>/<Vendor>_<Module>/templates/
2. <parent_theme_dir>/<Vendor>_<Module>/templates/
3. <grandparent_theme_dir>/<Vendor>_<Module>/templates/
4. <module_dir>/view/frontend/templates/ (module default)
⚡ Fallback Order for Static Files (CSS, JS, images): 
1. <theme_dir>/web/
2. <theme_dir>/<Vendor>_<Module>/web/
3. <parent_theme_dir>/web/
4. <parent_theme_dir>/<Vendor>_<Module>/web/
5. <module_dir>/view/frontend/web/ (module default)
6. lib/web/ (Magento library)
Example - Resolving product/view.phtml: 
Looking for: Magento_Catalog::product/view.phtml

Search order:
1. app/design/frontend/Acme/custom/Magento_Catalog/templates/product/view.phtml
2. app/design/frontend/Magento/blank/Magento_Catalog/templates/product/view.phtml
3. vendor/magento/module-catalog/view/frontend/templates/product/view.phtml ✓

First found = used

Overriding Templates

Override module template in theme: 
Module template location:
vendor/magento/module-catalog/view/frontend/templates/product/list.phtml

Override in theme:
app/design/frontend/Vendor/theme/Magento_Catalog/templates/product/list.phtml

Structure:
<theme_dir>/<Namespace>_<Module>/templates/<path/to/template.phtml>
Steps to override: 
1. Identify module template path
2. Create matching directory in theme
3. Copy template to theme (or create new)
4. Modify as needed
5. Clear cache: bin/magento cache:clean

⚠️ Important: Templates are NOT merged - first found wins entirely

Overriding Layout XML

Override vs Extend: 
Extend (default - files MERGE):
<theme_dir>/<Vendor>_<Module>/layout/catalog_product_view.xml

Override (complete replacement):
<theme_dir>/<Vendor>_<Module>/layout/override/base/catalog_product_view.xml
<theme_dir>/<Vendor>_<Module>/layout/override/<Parent_Vendor>_<parent_theme>/catalog_product_view.xml
Example - Extend layout (merge): 
File: app/design/frontend/Acme/theme/Magento_Catalog/layout/catalog_product_view.xml

<?xml version="1.0"?>
<page xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance">
    <body>
        <referenceBlock name="product.info" remove="true"/>
    </body>
</page>

Result: Merges with module layout + parent theme layout
Example - Override layout (replace): 
File: app/design/frontend/Acme/theme/Magento_Catalog/layout/override/base/catalog_product_view.xml

Result: Completely replaces module's catalog_product_view.xml
  • Extend: Layout files merge (most common)
  • Override base: Replace module layout entirely
  • Override theme: Replace parent theme layout

Overriding Static Files (CSS, JS, Images)

Override module static file: 
Module file:
vendor/magento/module-catalog/view/frontend/web/css/source/_module.less

Override in theme:
app/design/frontend/Vendor/theme/Magento_Catalog/web/css/source/_module.less
Theme-level static files: 
app/design/frontend/Vendor/theme/web/
├── css/
│   ├── source/
│   │   ├── _extend.less
│   │   └── _theme.less
│   ├── styles-m.less
│   └── styles-l.less
├── js/
│   └── theme.js
└── images/
    └── logo.svg

Static files also follow fallback: Current theme → Parent theme → Module → lib/web/

Module Fallback & Dependencies

When module files are used: 
Module files are the DEFAULT fallback when:
- No theme override exists
- No parent theme override exists

Modules define default UI components:
vendor/magento/module-catalog/
├── view/frontend/
│   ├── layout/
│   ├── templates/
│   └── web/
Disabling module output: 
// Disable module frontend output (keep functionality)
bin/magento module:disable Vendor_Module --clear-static-content

// or via layout XML
<page>
    <body>
        <referenceBlock name="block.name" remove="true"/>
    </body>
</page>

🔥 Key Exam Points:
  • Fallback is hierarchical: Theme → Parent → Module
  • Templates: First found wins (NO merge)
  • Layouts: ALL files merge (unless in override/ folder)
  • Static files follow same hierarchy
  • Override folders provide complete replacement
  • Developer mode: symlinks; Production: copies to pub/static/

Translation / i18n (Internationalization)

Magento uses CSV files and __() function for translations

Inline Translation in Templates (.phtml): 
<?php
// Basic translation
echo __('Add to Cart');

// Translation with variable
echo __('Price: %1', $price);

// Multiple variables
echo __('Product: %1, Price: %2', $productName, $price);

// Translation in attributes
?>
<button title="<?= $block->escapeHtmlAttr(__('Add to Cart')) ?>">
    <?= __('Add to Cart') ?>
</button>
Translation in Layout XML: 
<block name="custom.block">
    <arguments>
        <argument name="title" xsi:type="string" translate="true">My Title</argument>
    </arguments>
</block>

The translate="true" attribute marks the string for translation
Translation in JavaScript/Knockout: 
define(['jquery', 'mage/translate'], function($, $t) {
    // Using $t for translation
    var message = $t('Add to Cart');
    
    // With variable
    var message = $t('Price: %1').replace('%1', price);
    
    return {
        getMessage: function() {
            return $t('Hello World');
        }
    };
});
Translation CSV file structure: 
Location: 
- Module: <module_dir>/i18n/en_US.csv
- Theme: <theme_dir>/i18n/en_US.csv

Format: "Original String","Translated String"

Example en_US.csv:
"Add to Cart","Add to Cart"
"Price: %1","Price: %1"
"Product: %1, Price: %2","Product: %1, Price: %2"

Example de_DE.csv:
"Add to Cart","In den Warenkorb"
"Price: %1","Preis: %1"
"Product: %1, Price: %2","Produkt: %1, Preis: %2"
Translation Priority (fallback order): 
1. Database (Stores → Configuration → Translate Inline)
2. Theme translation (theme/i18n/en_US.csv)
3. Module translation (module/i18n/en_US.csv)
4. Magento core translation (lib/i18n/)
5. Original string (if no translation found)
Create translation dictionary: 
// For theme
app/design/frontend/Vendor/theme/i18n/en_US.csv
app/design/frontend/Vendor/theme/i18n/de_DE.csv

// For module
app/code/Vendor/Module/i18n/en_US.csv
app/code/Vendor/Module/i18n/de_DE.csv

After adding translations:
bin/magento cache:clean translate
bin/magento setup:static-content:deploy
Inline Translation in Admin: 
Stores → Configuration → Advanced → Developer → Translate Inline

Enable for Storefront: Yes
Enable for Admin: Yes

On frontend: Click text to edit translation (shown with red border)
Saves to database (highest priority)
Pluralization: 
// In template
<?= __('%1 item', $count) ?>
<?= __('%1 items', $count) ?>

// Better approach with ternary
<?= $count === 1 ? __('%1 item', $count) : __('%1 items', $count) ?>
🔥 Translation Exam Points:
  • Use __() function for translations in PHP/templates
  • Use $t() for translations in JavaScript
  • CSV format: "Original","Translation"
  • %1, %2 for variable placeholders
  • Always escape translated output: escapeHtml(__('text'))
  • Theme translations override module translations
  • Locale set by: Stores → Configuration → General → Locale

Creating a New Module

Modules extend Magento functionality. Understanding module structure is crucial.

Module Directory Structure: 
app/code/<Vendor>/<ModuleName>/
├── Api/                      // Service contracts
├── Block/                    // Block classes
├── Controller/               // Controllers
├── etc/
│   ├── module.xml           // Module declaration (REQUIRED)
│   ├── di.xml               // Dependency injection
│   ├── acl.xml              // Access Control List
│   └── frontend/
│       └── routes.xml       // Frontend routes
├── Helper/                   // Helper classes
├── Model/                    // Models & Resource Models
├── Observer/                 // Event observers
├── Plugin/                   // Plugins (interceptors)
├── Setup/                    // Install/upgrade scripts
├── ViewModel/                // View models
├── view/
│   └── frontend/
│       ├── layout/          // Layout XML
│       ├── templates/       // .phtml templates
│       ├── web/
│       │   ├── css/         // CSS/LESS
│       │   ├── js/          // JavaScript
│       │   └── images/      // Images
│       └── requirejs-config.js
├── registration.php         // Module registration (REQUIRED)
└── composer.json            // Composer config
Step 1: Create registration.php (REQUIRED): 
// app/code/Vendor/ModuleName/registration.php
<?php
use Magento\Framework\Component\ComponentRegistrar;

ComponentRegistrar::register(
    ComponentRegistrar::MODULE,
    'Vendor_ModuleName',
    __DIR__
);
Step 2: Create etc/module.xml (REQUIRED): 
// app/code/Vendor/ModuleName/etc/module.xml
<?xml version="1.0"?>
<config xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" 
        xsi:noNamespaceSchemaLocation="urn:magento:framework:Module/etc/module.xsd">
    <module name="Vendor_ModuleName" setup_version="1.0.0">
        <sequence>
            <!-- Dependencies (load after these modules) -->
            <module name="Magento_Catalog"/>
            <module name="Magento_Checkout"/>
        </sequence>
    </module>
</config>
Step 3: Create composer.json (optional but recommended): 
{
    "name": "vendor/module-name",
    "description": "Custom Module",
    "type": "magento2-module",
    "version": "1.0.0",
    "require": {
        "php": "~8.1.0||~8.2.0||~8.3.0",
        "magento/framework": "103.0.*"
    },
    "autoload": {
        "files": ["registration.php"],
        "psr-4": {
            "Vendor\\ModuleName\\": ""
        }
    }
}
Step 4: Enable the module: 
bin/magento module:enable Vendor_ModuleName
bin/magento setup:upgrade
bin/magento setup:di:compile
bin/magento setup:static-content:deploy (if production)
bin/magento cache:clean
Step 5: Create frontend routes (if needed): 
// etc/frontend/routes.xml
<?xml version="1.0"?>
<config xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" 
        xsi:noNamespaceSchemaLocation="urn:magento:framework:App/etc/routes.xsd">
    <router id="standard">
        <route id="customroute" frontName="customroute">
            <module name="Vendor_ModuleName"/>
        </route>
    </router>
</config>

Access: yourdomain.com/customroute/controller/action
Example Controller: 
// Controller/Index/Index.php
<?php
namespace Vendor\ModuleName\Controller\Index;

use Magento\Framework\App\Action\Action;
use Magento\Framework\App\Action\Context;
use Magento\Framework\View\Result\PageFactory;

class Index extends Action
{
    protected $pageFactory;

    public function __construct(
        Context $context,
        PageFactory $pageFactory
    ) {
        parent::__construct($context);
        $this->pageFactory = $pageFactory;
    }

    public function execute()
    {
        return $this->pageFactory->create();
    }
}
Example Block: 
// Block/CustomBlock.php
<?php
namespace Vendor\ModuleName\Block;

use Magento\Framework\View\Element\Template;

class CustomBlock extends Template
{
    public function getCustomData()
    {
        return 'Hello from custom module!';
    }
}
Useful Commands: 
bin/magento module:status                    # Check module status
bin/magento module:enable Vendor_ModuleName  # Enable module
bin/magento module:disable Vendor_ModuleName # Disable module
bin/magento module:uninstall Vendor_ModuleName # Uninstall (if setup script exists)
🔥 Exam Critical Points:
  • Two REQUIRED files: registration.php & etc/module.xml
  • Module naming: Vendor_ModuleName (PSR-4)
  • <sequence> defines load order dependencies
  • Always run setup:upgrade after creating/modifying module
  • Modules in app/code/ or vendor/ (composer)
